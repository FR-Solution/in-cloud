# Регламент
## Написанию документации по сертификатам

### Введение
Данный модуль документации написан, для быстрого и качественного описания документации, где требется описывать сертификаты приложений.

### Структура
```bash
.
├── centerAuthority
|   └── main.mdx
│   └── caName
│       ├── bar.mdx
│       ├── kubeadm.mdx
│       ├── main.mdx
│       └── openssl.mdx
└── components
|    └── componentName
|        └── main.mdx
|        └── certificateDirection
|            ├── bar.mdx
|            ├── kubeadm.mdx
|            ├── main.mdx
|            └── openssl.mdx
└── certificateAuthorityCerts.mdx
└── componentsCerts.mdx

```

| Поле                                                            | Тип         | Описание                                                                                                                                                      |
| :-------------------------------------------------------------- | :---------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| centerAuthority                                                 | директория  | Хранение инструкций по жизненному циклу ЦА                                                                                                                    |
| centerAuthority/$caName                                      | директория  | Хранение инструкций конкретного ЦА                                                                                                                            |
| centerAuthority/$caName/*.mdx                                | файл        | Инструкция жизненного цикла ЦА для конкретного инструмента, например OpenSSL                                                                                  |
| centerAuthority/$caName/main.mdx                             | файл        | Основная инструкция жизненного цикла ЦА, объединяющая файлы `centerAuthority/$caName/*.mdx`                                                                  |
| centerAuthority/main.mdx                                        | файл        | Описывающий все сертификаты компонентов, описанные в `centerAuthority/$caName/main.mdx `                                                                     |
| components                                                      | директория  | Хранение папок компонентов продукта (по одной папке на компонент)                                                                                             |
| components/$componentName                                    | директория  | Хранение инструкций по жизненному циклу сертификатов конкретного компонента                                                                                   |
| components/main.mdx                                             | файл        | Описывающий все сертификаты компонентов, описанные в `components/$componentName/$certificateDirection/*` (данный файл имеем итоговую структуру документа) |
| components/$componentName/$certificateDirection           | директория  | Хранение инструкций для конкретного назначения сертификата в компоненте, например etcdClient или kubernetesServer                                             |
| components/$componentName/$certificateDirection/main.mdx  | файл        | Основная инструкция жизненного цикла сертификата компонента, объединяющая файлы `components/$componentName/$certificateDirection/*.mdx `                  |
| components/$componentName/$certificateDirection/*.mdx     | файл        | Инструкция жизненного цикла сертификата компонента для конкретного инструмента, например OpenSSL                                                              |
| certificateAuthorityCerts.mdx                                   | файл        | Описывающий все сертификаты ЦА, описанные в `centerAuthority/main.mdx` (данный файл имеем итоговую структуру документа)                                         |
| componentsCerts.mdx                                             | файл        | Описывающий все сертификаты компонентов, описанные в `components/main.mdx` (данный файл имеем итоговую структуру документа)                                     |


1. *$caName*:

   - Имя должно быть записано в стиле `camelCase` (также известном как `dromedaryCase`).
   - Имя должно быть интуитивно понятным и отражать суть.

2. *$componentName*:

   - Имя должно быть записано в стиле `camelCase` (также известном как `dromedaryCase`).
   - Имя должно быть интуитивно понятным и отвечать на вопрос: "Что это за компонент?"

3. *$certificateDirection*:

   - Имя должно быть записано в стиле `camelCase` (также известном как `dromedaryCase`).
   - Имя должно быть интуитивно понятным и отвечать на вопрос: "Для чего используется данный сертификат?"

4. *Файлы с расширением *.mdx*:

   - Имена файлов должны быть записаны в нижнем регистре (`lowercase`).
   - Имена файлов должны соответствовать исходному названию инструмента.


### Правила к описанию сертификата

1. **Используйте формат MDX (Markdown с поддержкой JSX):**
- Это позволяет комбинировать обычный Markdown-текст с компонентами React и JSX-синтаксисом для более гибкой презентации контента.

2. **Импортируйте необходимые модули и константы в начале документа:**
- Импортируйте константы, используемые в документе, из соответствующих путей:
```jsx
import { CERTIFICATES } from '@site/src/constants/kubernetes/certs';
import { CUSTOM_VALUE } from '@site/src/constants/kubernetes/customValue';
```
- Импортируйте компоненты и функции, необходимые для отображения и форматирования контента:
```jsx
import CodeBlock from '@theme/CodeBlock';
import dedent from 'dedent';
```

3. **Структурируйте документ с помощью заголовков:**
- Используйте заголовки третьего уровня (`###`) для основных разделов документа.
- Используйте заголовки четвертого уровня (`####`) для подразделов или шагов внутри основных разделов.
- Пример:
```markdown
### ETCD client certificate
#### Переменные окружения openssl для сертификата `kubernetes-etcd-client`
```

4. **Используйте компонент `` для отображения блоков кода с подсветкой синтаксиса:**
- Укажите язык программирования или формат внутри атрибута `language`, например `language="bash"`.
- Для многострочных строк используйте функцию `dedent` для удаления лишних отступов и сохранения форматирования:
```jsx

{dedent`
# Ваш код здесь
`}

```

5. **Интегрируйте переменные из импортированных констант в код с использованием шаблонных строк:**
- При объявлении переменных или в командах используйте синтаксис `${VARIABLE}` для вставки значений:
```bash
export KUBERNETES_ETCD_CLIENT_CN="${CERTIFICATES.kubernetesEtcdClient.cname}"
```

6. **Предоставляйте пошаговые инструкции с соответствующими заголовками и описаниями:**
- Каждый шаг процесса (например, генерация ключа, создание CSR, подпись сертификата) должен быть выделен отдельным подразделом.
- Начинайте каждый шаг с пояснительного заголовка и, при необходимости, краткого описания.

7. **Для команд, выполняемых в терминале, используйте соответствующие блоки кода:**
- Если команда короткая или не требует дополнительного форматирования, можно использовать стандартный Markdown-блок кода с указанием языка:
```bash
openssl genrsa -out key.pem 2048
```
- Для более сложных скриптов используйте `` с `dedent` для сохранения структуры и отступов.

8. **При необходимости создавайте файлы конфигурации прямо в документе с помощью heredoc-синтаксиса:**
- Используйте `cat < filename` для генерации файлов конфигурации:
```bash
cat < ${KUBERNETES_ETCD_CLIENT_CRT_CONF}
[ req ]
# Конфигурация здесь
EOF
```

9. **Сохраняйте консистентный стиль оформления throughout the document:**
- Отступы, пустые строки и форматирование должны быть последовательными.
- Комментарии внутри кодовых блоков должны быть понятными и полезными.

10. **Объясняйте назначение каждого шага или блока кода, если это необходимо для понимания:**
- Хотя основной акцент на коде, краткие пояснения помогают читателю понять контекст и цель команд.
- Например:
```markdown
#### Генерация приватного ключа сертификата kubernetes-etcd-client
```

11. **Не добавляйте лишней информации или шагов, которые не отражены в исходном документе:**
- Следуйте строго содержанию предоставленного документа, не выдумывая дополнительных деталей.

12. **Обеспечьте корректность и работоспособность всех команд и скриптов:**
- Код должен быть проверен на синтаксические ошибки.
- Переменные и пути должны соответствовать тем, что используются в импортированных константах.

13. **Используйте английский язык для технических терминов и названий файлов/переменных, но русские заголовки и описания:**
- Это соответствует стилю исходного документа, где технические элементы остаются на английском, а объяснения на русском.

14. **Избегайте избыточности и повторений:**
- Если определенные команды или шаги повторяются, рассмотрите возможность вынести общую информацию в отдельный раздел или использовать ссылки.

15. **В конце документа не добавляйте лишние символы или некорректные окончания:**
- Убедитесь, что документ завершается корректно, без лишних знаков или символов, которые могут привести к ошибкам парсинга.
