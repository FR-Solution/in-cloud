## Введение

Данная тема содержит в себе описание подключения к БД etcd (Используемой для KubeAPI), процессов отладки, создания "мгновенных снимков", восстановления БД и полезных команд.

*****

## Пререквизиты

### С чего начать

Рекомендуем разбить процесс отладки на следующие этапы:

1. Проверка мониторинга. #TODO ссылка на страницу с графаной
2. Состояние подов кластера ETCD.
3. Сбор и настройка пререквизитов.
4. Подключение к кластеру ETCD.
5. Оценка состояния кластера ETCD и его узлов.
6. Отладка инфраструктуры и сопутствующих служб (При необходимости).
7. Создание моментального снимка и резервной копии БД.
8. Восстановление кластера (При возможности).
9. Восстановление кластера в режиме одного узла.
10. Восстановление данных в БД из резервной копии или моментального снимка.
11. Подключение остальных узлов кластера.

### Подготовка окружения

#### Скачивание etcdctl

Для выполнения операций с файлами ETCD кластера, в условиях отсутствия работающих контейнеров ETCD, мы будем использовать локальный исполняемый файл etcdctl (Далее `локальный etcdctl`).
Etcdctl можно получить из опубликованных релизов etcd (Например версия <code>{ETCD_version}</code>).
Выполните следующие команды, чтобы скачать релиз, распаковать и извлечь из него etcdctl в директорию <code>{ETCD_workdir}</code>:

<CodeBlock language="bash">
  {dedent`
    export DOWNLOAD_URL=https://github.com/etcd-io/etcd/releases/download
    export ETCD_VER=${ETCD_version}
    export ETCD_WORKDIR="${ETCD_workdir}"

    curl -L \${DOWNLOAD_URL}/\${ETCD_VER}/etcd-\${ETCD_VER}-darwin-amd64.zip \
      -o etcd-\${ETCD_VER}-darwin-amd64.zip

    unzip etcd-\${ETCD_VER}-darwin-amd64.zip etcd-\${ETCD_VER}-darwin-amd64/etcdctl \
      && rm -f etcd-\${ETCD_VER}-darwin-amd64.zip

    mkdir ${ETCD_workdir}

    mv etcd-\${ETCD_VER}-darwin-amd64/etcdctl ${ETCD_workdir}/etcdctl \
      && rm -rf etcd-\${ETCD_VER}-darwin-amd64

    chmod +x ${ETCD_workdir}/etcdctl

    ${ETCD_workdir}/etcdctl version
  `}
</CodeBlock>

#### Подключение к узлу K8s кластера.

Задайте имя кластера в переменную `K8S_CLUSTER_NAME`.

Определите адреса управляющих узлов K8s кластера и подключитесь к одному из них.
Сделать это можно через запрос к инфраструктурному кластеру:

```bash
kubectl get machine -n ${K8S_CLUSTER_NAME} \
  -l cluster.x-k8s.io/control-plane-name \
  -o custom-columns=Name:.metadata.name,ADDRESS:.status.addresses\[*\].address
```

<details>

<CodeBlock language="bash">
{`

scp ${ETCD_workdir}/etcdctl capv@\${K8S_NODE_ADDRESS_1}:${ETCD_workdir}/etcdctl
ssh capv@${ETCD_k8s_node_address_1}
sudo su -
`}
</CodeBlock>

## Подключение к etcd

Определив пререквизиты, настроим псевдоним для упрощенного подключения к ETCD кластеру.

<Tabs groupId="install-type">
  <TabItem value='Локальный etcdctl'>
    Определим псевдоним `etcdctl` на локальный файл с указанием сертификатов:

    <CodeBlock language="bash">
      {dedent`
        alias etcdctl=\'${ETCD_workdir}/etcdctl \
          --cert \${PEER_CERT_PATH} \
          --key \${PEER_KEY_PATH} \
          --cacert \${CA_CERT_PATH}\'
      `}
    </CodeBlock>

  </TabItem>
  <TabItem value='crictl с мастера'>
    Т.к. будет использоваться `etcdctl` находящийся внутри etcd контейнера, для начала нужно получить его `CONTAINER_ID`.
    Затем создадим псевдоним `ectl` (чтобы не путать его с `etcdctl` на мастере) и будем актуализировать данные по текущему контейнеру в каждом вызове псевдонима.
    Также укажем путь до сертификатов.

    ```bash
    alias ectl='crictl exec
      "$(crictl ps -a  \
        --label io.kubernetes.container.name=etcd \
        --label io.kubernetes.pod.namespace=kube-system \
        | awk 'NR>1{r=$1} $0~/Running/{exit} END{print r}')" \
      etcdctl \
      --cert ${PEER_CERT_PATH} \
      --key ${PEER_KEY_PATH} \
      --cacert ${CA_CERT_PATH}'
    ```

  </TabItem>
  <TabItem value='nsenter с мастера'>
    Т.к. будет использоваться `etcdctl` находящийся внутри пространства имен процесса на котором запущен контейнер, для начала нужно получить `CONTAINER_ID`.
    Затем на его основе можно получить `PID`.
    Таким образом создадим псевдоним `ectl` (чтобы не путать его с `etcdctl` на мастере) и будем актуализировать данные по текущему контейнеру и процессу в каждом вызове псевдонима.
    Также укажем путь до сертификатов.

    ```bash
    alias etcdctl='nsenter \
      -m -n -t $(crictl inspect \
      "$(crictl ps -a  \
        --label io.kubernetes.container.name=etcd \
        --label io.kubernetes.pod.namespace=kube-system \
        | awk 'NR>1{r=$1} $0~/Running/{exit} END{print r}')" \
      | jq .info.pid) \
      --all etcdctl --cert ${PEER_CERT_PATH} \
      --key ${PEER_KEY_PATH} \
      --cacert ${CA_CERT_PATH}'
    ```

  </TabItem>
  <TabItem value='kubectl exec'>
    Зная имя пода и владея доступом до KubeAPI, можем использовать псевдоним `kectl` (чтобы не путать его с `etcdctl` и `ectl` на мастере) описанный таким образом:

    ```bash
    alias kectl='kubectl \
      exec -it \
      -n kube-system \
      ${ETCD_POD_1} -- \
      etcdctl --cert ${PEER_CERT_PATH} \
      --key ${PEER_KEY_PATH} \
      --cacert ${CA_CERT_PATH}'
    ```

  </TabItem>
</Tabs>

### Восстановление кластера

#### Восстановление из мгновенных снимков



<Tabs groupId="install-type">
  <TabItem value='Master-1'>
    0) Загрузите файл мгновенного снимка на ВМ  первого узла кластера по пути <code>{ETCD_workdir}/backup.db</code>

    1) Подключитесь к ВМ первого узла кластера по ssh и остановите `kubelet`, чтобы не позволить ему запускать ETCD.

    ```bash
    systemctl stop kubelet
    ```

    2) Остановите и удалите контейнер `etcd`:

    ```bash
    export CONTAINER_ID="$(crictl ps -a  \
        --label io.kubernetes.container.name=etcd \
        --label io.kubernetes.pod.namespace=kube-system \
        | awk 'NR>1{r=$1} $0~/Running/{exit} END{print r}')"
    crictl stop ${CONTAINER_ID}
    crictl rm ${CONTAINER_ID}
    ```

    3) Создайте (бэкап)[создание-резервной-копии-файлов-бд].

    4) Удалите старые файлы из директории <code>{ETCD_data_dir}</code>

    <CodeBlock language="bash">
      {`
        rm -rf ${ETCD_workdir}/*
      `}
    </CodeBlock>

    5) Подготовьте БД из снимка:
    <CodeBlock language="bash">
      {dedent`
        etcdctl snapshot restore ${ETCD_workdir}/backup.db \
          --name=\${K8S_NODE_1} \
          --data-dir ${ETCD_data_dir} \
          --initial-advertise-peer-urls=https://\${K8S_NODE_ADDRESS_1}:2380 \
          --initial-cluster=\${K8S_NODE_1}=https://\${K8S_NODE_ADDRESS_1}:2380
      `}
    </CodeBlock>

    6) Запустите kubelet:

    ```bash
    systemctl start kubelet
    ```

    7) Проверьте состояние контейнера и наличие ошибок

Проверьте размер файла `/var/lib/etcd/member/snap/db`. Если его размер превышает 2 GB рекомендуем запустить дефрагментацию.
