## Введение

Данная тема содержит в себе описание подключения к БД etcd (Используемой для KubeAPI), процессов отладки, создания "мгновенных снимков", восстановления БД и полезные команды.

## Пререквизиты

### С чего начать

Рекомендуем разбить процесс отладки на следующие этапы:

1. Проверка мониторинга. #TODO ссылка на страницу с графаной
2. Состояние подов кластера ETCD.
3. Сбор и настройка пререквизитов.
4. Подключение к кластеру ETCD.
5. Оценка состояния кластера ETCD и его узлов.
6. Отладка инфраструктуры и сопутствующих служб (При необходимости).
7. Создание моментального снимка и резервной копии БД.
8. Восстановление кластера (При возможности).

10. Восстановление данных в БД из резервной копии или моментального снимка.
11. Подключение остальных узлов кластера.

### Подготовка окружения

#### Скачивание etcdctl

Для выполнения операций с файлами ETCD кластера, в условиях отсутствия работающих контейнеров ETCD, мы будем использовать локальный etcdctl файл.
Исполняемый файл etcdctl можно получить из опубликованных релизов etcd (Например версия v3.5.17).
Выполните следующие команды, чтобы скачать релиз, распаковать и извлечь из него etcdctl:

```bash
DOWNLOAD_URL=https://github.com/etcd-io/etcd/releases/download
ETCD_VER=v3.5.17

curl -L ${DOWNLOAD_URL}/${ETCD_VER}/etcd-${ETCD_VER}-darwin-amd64.zip \
  -o etcd-${ETCD_VER}-darwin-amd64.zip
unzip etcd-${ETCD_VER}-darwin-amd64.zip etcd-${ETCD_VER}-darwin-amd64/etcdctl \
  && rm -f etcd-${ETCD_VER}-darwin-amd64.zip
mv etcd-${ETCD_VER}-darwin-amd64/etcdctl /tmp/etcdctl \
  && rm -rf etcd-${ETCD_VER}-darwin-amd64

/tmp/etcdctl version
```

## Подключение к etcd

Определив пререквизиты, настроим псевдоним для упрощенного подключения к ETCD кластеру.

<Tabs groupId="install-type">
  <TabItem value='etcdctl с мастера'>
  Определим alias etcdctl на локальный файл с указанием сертификатов:

  ```
  alias etcdctl='/tmp/etcdctl \
    --cert ${PEER_CERT_PATH} \
    --key ${PEER_KEY_PATH} \
    --cacert ${CA_CERT_PATH}'
  ```

  </TabItem>
  <TabItem value='crictl с мастера'>
  Т.к. будет использоваться etcdctl находящийся внутри etcd контейнера, для начала нужно получить его CONTAINER_ID.
  Затем создадим псевдоним ectl (чтобы не путаться его с etcdctl на мастере) и будем актуализировать данные по текущему контейнеру в каждом вызове псевдонима.
  Также укажем путь до сертификатов.

  ```bash
  alias ectl='crictl exec
    "$(crictl ps -a  \
      --label io.kubernetes.container.name=etcd \
      --label io.kubernetes.pod.namespace=kube-system \
      | awk 'NR>1{r=$1} $0~/Running/{exit} END{print r}')" \
    etcdctl \
    --cert ${PEER_CERT_PATH} \
    --key ${PEER_KEY_PATH} \
    --cacert ${CA_CERT_PATH}'

  ```

  </TabItem>
  <TabItem value='nsenter с мастера'>
  Т.к. будет использоваться etcdctl находящийся внутри пространства имен процесса на котором запущен контейнер, для начала нужно получить CONTAINER_ID.
  Затем на его основе можно получить PID;
  Таким образом создадим псевдоним ectl (чтобы не путаться его с etcdctl на мастере) и будем актуализировать данные по текущему контейнеру и процессу в каждом вызове псевдонима.
  Также укажем путь до сертификатов.

  ```bash
  alias etcdctl='nsenter \
  -m -n -t $(crictl inspect \
  "$(crictl ps -a  \
    --label io.kubernetes.container.name=etcd \
    --label io.kubernetes.pod.namespace=kube-system \
    | awk 'NR>1{r=$1} $0~/Running/{exit} END{print r}')" \
  | jq .info.pid) \
  --all etcdctl --cert /etc/kubernetes/pki/etcd/peer.crt \
  --key /etc/kubernetes/pki/etcd/peer.key \
  --cacert /etc/kubernetes/pki/etcd/ca.crt \
  --endpoints=$ENDPOINTS'
  ```

  </TabItem>
  <TabItem value='kubectl exec'>
  Зная имя пода и владея доступом до kube-api, можем использовать такой псевдоним:

  ```bash
  alias etcdctl='kubectl \
  exec -it \
  -n kube-system \
  ${ETCD_POD_1} -- \
  etcdctl --cert /etc/kubernetes/pki/etcd/peer.crt \
  --key /etc/kubernetes/pki/etcd/peer.key \
  --cacert /etc/kubernetes/pki/etcd/ca.crt \
  --endpoints=$ENDPOINTS'
  ```

  </TabItem>

## Полезные команды

Для использования исполняемого файла на мастере будем использовать псевдоним `etcdctl`, для использования etcdctl внутри контейнера, используем псевдоним `ectl`.

| Команды | описание |
| :--- | :--- |
| `etcdctl --endpoints=${ENDPOINT_1} member list -w table` | Вывести перечень узлов ETCD кластера с одного из узлов в табличном формате |
| `ectl member list -w table` | Вывести перечень узлов ETCD кластера с одного из узлов в табличном формате |
| `etcdctl --endpoints=${ENDPOINTS} endpoint status -w table` | Вывести статус узлов ETCD кластера в табличном формате |
| `ectl --cluster member list -w table` | Вывести статус узлов ETCD кластера в табличном формате |
| `etcdctl --endpoints=${ENDPOINTS} get prefix /` | Вывести все ключи и значения, чей ключ удовлетворяет заданному префиксу. Возможна комбианация с флагамаи `--keys-only` и `--print-value-only` |
| `ectl --cluster get prefix /` | Вывести все ключи и значения, чей ключ удовлетворяет заданному префиксу. Возможна комбианация с флагамаи `--keys-only` и `--print-value-only` |
| `etcdctl --endpoints=${ENDPOINTS} snapshot save PATH_TO_SAVE.db` | Создать моментальный снимок состояния базы данных ETCD (Снимок будет сохранен на машину с которой был запущен etcdctl) |
| `ectl --cluster snapshot save PATH_TO_SAVE.db` | Создать моментальный снимок состояния базы данных ETCD (Снимок будет сохранен внутри контейнера)  |
| `ectl defrag` | Выполнить дефрагментацию файлов БД. (При выключенном кластере применяется с флагом --data-dir и указыванием пути до файла)  |
| `ectl --endpoints=${ENDPOINT_1} defrag` | Выполнить дефрагментацию файлов БД. (При выключенном кластере применяется с флагом --data-dir и указыванием пути до файла)  |
| `ectl --endpoints=${ENDPOINT_1} companation 1` | Выполнить компоновку данных и назначить номер ревизии |
| `ectl --cluster companation 1` | Выполнить компоновку данных и назначить номер ревизии |


`ectl --cluster get /registry --prefix --keys-only | grep -v ^$ | awk -F '/'  '{ h[$3]++ } END {for (k in h) print h[k], k}' | sort -nr`
посчитать количество ключей по корневым префикасам.
