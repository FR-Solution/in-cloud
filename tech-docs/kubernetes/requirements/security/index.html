2.10 Информационная безопасность

2.10.1 Требования к идентификации и аутентификации​

№Требование
1Обеспечивается создание уникальной учетной записи для каждого пользователя
2Механизмы аутентификации реализуются с использованием защищенных протоколов аутентификации. При хранении и передаче конфиденциальность паролей обеспечивается шифрованием или хешированием с применением стойких криптографических алгоритмов
3Обеспечивается возможность аутентификации в Active Directory
4Реализуются механизмы двухфакторной аутентификации пользователей
5Проверка учетных данных для всех учетных записей проводится на стороне серверных компонентов ИС (отсутствие возможности оффлайн аутентификации)
6В процессе аутентификации в ИС пароль не отображается при его вводе, проверка введенной информации (логин, пароль) осуществляется только после полного ее ввода, в случае обнаружения ошибки, система не уточняет, какие именно данные введены неправильно

2.10.2 Сбор и анализ событий информационной безопасности​

№Требование
1Обеспечивается возможность регистрации событий безопасности, включая: попытки идентификации, изменения полномочий, модификацию профиля пользователей, действия привилегированных пользователей, доступ к защищаемым объектам
2Обеспечивается фиксация информации для каждого события безопасности: тип события, дата и время, идентификационная информация, результат события, субъект и объект доступа, описание действий
3Обеспечивается ограничение доступа к журналам регистрации событий безопасности только уполномоченным пользователям

2.10.3 Защита информации при использовании мобильных устройств​

№Требование
1Ограничение доступа пользователей с использованием разрешенных версий мобильного ПО
2Применение технологии SSL/TLS Pinning для защиты корневых сертификатов
3Использование защищенных хранилищ для хранения аутентификационных данных пользователя
4Журналирование событий в незащищенные области не допускается
5Ограничение передачи конфиденциальной информации между приложениями

2.10.4 Требования к парольной защите и учетным записям​

№Требование
1Пользователям обеспечивается возможность самостоятельной установки и смены пароля
2Обеспечивается возможность назначения первичного пароля администратором ИС и обязательной смены пароля пользователем при первичной аутентификации в ИС
3Неиспользуемая в течение 30 (тридцать) дней учетная запись пользователя блокируется в ИС
4Обеспечивается возможность заблаговременного (не менее чем за 14 дней) оповещения пользователей о необходимости смены пароля (посредством сообщений, подсказок или почтовых рассылок)
5Реализована возможность установки срока длительности простоя (не более 15 минут) пользовательской сессии, после которого сессия принудительно блокируется до повторной аутентификации
6Обеспечивается возможность хранения истории паролей (хэшей паролей) пользователей для предотвращения повторного их использования. Проверяется повтор последних 7 паролей
7Обеспечивается возможность установки ограничений на длину и сложность пароля (буквы в верхнем и нижнем регистрах, цифры и специальные символы)
8Обеспечивается возможность блокировки учетной записи на заранее определенный срок после заданного количества неудачных попыток аутентификации
9Обеспечивается принудительная смена пароля через установленный промежуток времени
10Реализованы механизмы усложняющих техник ввода пароля: CAPTCHA, увеличение времени ожидания

2.10.5 Требования к управлению правами доступа/разделению полномочий​

№Требование
1Обеспечивается реализация механизмов авторизации пользователей с разделением прав доступа к информации/данным и функциям внутри ИС
2Обеспечивается предоставление пользователям прав доступа на основании атрибутной или ролевой модели
3Обеспечивается явное ограничение или запрет на действия пользователя в ИС до прохождения процедур идентификации и аутентификации
4Запрещён по умолчанию прямой доступ пользователей, администраторов, третьих лиц в продуктивные базы данных ИС
5Хранение и обработка информации Банка во внешних облачных сервисах запрещена
6Осуществляется информирование пользователей ИС перед непосредственным получением доступа к ИС / ИА в формате системного уведомления / баннера. Уведомление или баннер сохраняются на экране, пока пользователи не подтвердят условия использования и примут явные меры для входа в ИС

2.10.6 Общие требования к информационной безопасности​

№Требование
1Просмотр сводной информации по политикам безопасности в кластере, в первую очередь Network Policy
2Поддерживает разграничение доступа на основании ролевой (RBAC) или атрибутивной (ABAC) модели
3Поддерживает разграничение доступа на уровне записей/сущностей одного типа (row-level security)
4Имеется механизм шифрования хранимых данных (указать, каких именно данных и требования к алгоритмам шифрования)
5Аудит событий по изменению конфигураций, настроек информационной системы, установки обновлений
6Аудит событий по изменению данных в результате выполнения бизнес-процессов и хронологическую последовательность действий пользователей
7Возможность исключать из записи в журнал событий чувствительные данные, такие как PAN карт и т.п.
